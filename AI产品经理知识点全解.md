## 第 1 层：大模型基础

### 1.1 LLM（大语言模型）是什么
- **本质**：根据前面的文字，预测下一个最可能的字的概率机器
- **不是数据库**：它不"知道"事实，只是"接话接得像"
- **训练过程**：用海量文本训练，学会语言的统计规律
- **为什么重要**：理解这个本质，才能理解为什么会幻觉、为什么需要 RAG

### 1.2 Token（词元）
- **定义**：模型处理文本的最小单位
- **中英文差异**：
  - 英文：1 个单词约 1 个 token（"hello" = 1 token）
  - 中文：1 个汉字约 1～2 个 token（"你好" = 2～3 tokens）
- **为什么重要**：计费、上下文长度、限流，全部按 token 算
- **实际应用**：估算成本必须懂 token 计算

### 1.3 上下文窗口（Context Window）
- **定义**：模型一次能"看到"的 token 上限
- **常见窗口大小**：
  - GPT-3.5: 4K / 16K
  - GPT-4: 8K / 32K / 128K
  - Claude: 100K / 200K
  - Gemini: 1M
- **限制**：输入 + 输出总和不能超过窗口
- **影响**：窗口越大越贵、越慢
- **RAG 的必要性**：窗口装不下全部知识库，所以要先检索再塞进去

### 1.4 预训练 vs 微调
- **预训练（Pre-training）**：
  - 出厂时用海量通用文本训练
  - 获得通用语言能力
  - 用户无法控制
- **微调（Fine-tuning）**：
  - 用你自己的数据再训一遍
  - 改变模型的"性格/专长"
  - 成本：需要数据标注、GPU、时间
  - **决策原则**：Prompt → RAG → 微调（成本递增）

### 1.5 参数量
- **含义**：模型的大小（7B = 70 亿参数、72B = 720 亿参数）
- **权衡**：
  - 参数越多 → 越聪明、越贵、越慢
  - 参数越少 → 越快、越便宜、效果差
- **产品决策**：根据场景选合适规模的模型

## 第 2 层：调用参数（API 调用的旋钮）

### 2.1 Temperature（温度）⭐⭐⭐
- **作用**：控制输出的随机性 / 创造性
- **取值范围**：0 ～ 2（常用 0 ～ 1）
- **效果**：
  - `temperature = 0`：几乎确定，同样问题每次答得几乎一样
    - 适合：客服、事实问答、信息抽取
  - `temperature = 0.3`：略有变化，但仍然保守
    - 适合：智能客服（你第一个项目就是这个）
  - `temperature = 0.7～1.0`：发散、有创意，但容易跑偏
    - 适合：文案创作、起名、头脑风暴
  - `temperature > 1.0`：很随机，可能胡言乱语
- **产品决策案例**：
  - 客服机器人：0.1～0.3（要稳定、少幻觉）
  - 营销文案生成：0.7～0.9（要创意）
  - 代码生成：0.2（要精确）

### 2.2 Top-p（核采样）
- **作用**：另一种控制随机性的方式
- **原理**：只在"累计概率前 p"的候选词里挑
- **例子**：`top_p = 0.9` 表示只考虑累计概率达到 90% 的词
- **与 temperature 的关系**：一般只调一个，不要俩一起猛调
- **推荐设置**：
  - 确定性任务：top_p = 0.9，temperature = 0.1
  - 创造性任务：top_p = 0.95，temperature = 0.8

### 2.3 Top-k
- **作用**：只在概率最高的 k 个词里挑
- **与 top_p 的区别**：
  - top_k：固定数量
  - top_p：累计概率阈值
- **使用建议**：一般用 top_p，更灵活

### 2.4 max_tokens（最大输出长度）
- **作用**：限制回答最多生成多少 token
- **影响**：
  - 设太小：答案被截断
  - 设太大：费钱、浪费
- **产品决策**：根据实际需求设置合理上限

### 2.5 frequency_penalty / presence_penalty
- **frequency_penalty**：惩罚已经出现多次的词
- **presence_penalty**：惩罚已经出现过的词（不管几次）
- **作用**：防止模型"车轱辘话"
- **取值**：-2.0 ～ 2.0，正值表示惩罚

### 2.6 stop（停止词）
- **作用**：遇到指定字符串就停止输出
- **应用**：结构化场景，比如让模型输出到特定分隔符就停

### 2.7 seed（随机种子）
- **作用**：固定随机种子，让"随机"可复现
- **应用**：测试、调试、A/B 对比实验

### 2.8 stream（流式输出）
- **两种模式**：
  - `stream = False`：等全部生成完再一次性返回
  - `stream = True`：像打字机一样一个字一个字蹦
- **产品决策**：
  - 流式：首字延迟低，体验好，但实现复杂
  - 非流式：实现简单，但用户要等很久才看到第一个字

### 2.9 角色（role）
- **system**：设定 AI 的人设、规则、边界
- **user**：用户说的话
- **assistant**：AI 之前说的话
- **多轮对话**：就是把这三种角色的消息按时间顺序拼起来

## 第 3 层：Prompt 工程

### 3.1 System Prompt（系统提示词）
- **作用**：给 AI 定身份、定规则、定边界
- **重要性**：降低幻觉的第一道闸
- **示例**（你第一个项目）：
  ```
  你是 SmartGuide 客服助手，专门回答智能硬件使用问题。
  回答规则：
  1. 只根据提供的文档内容回答，不能编造信息
  2. 如果文档中没有相关信息，明确告知"该信息不在设备使用指南中，无法作答"
  ```

### 3.2 Zero-shot / Few-shot
- **Zero-shot**：不给例子，直接让模型做任务
- **Few-shot**：给几个例子，让模型学习模式
- **效果**：Few-shot 通常能显著提升格式和准确度
- **示例**：
  ```
  问题：智能手环如何配对？
  答案格式：
  1. 打开手环
  2. 打开 APP
  3. 点击配对
  
  问题：智能音箱如何联网？
  答案格式：（模型学会了这个格式）
  ```

### 3.3 Chain-of-Thought（思维链 CoT）
- **原理**：让模型"一步步想"
- **触发方式**：在 prompt 里加"让我们一步步思考"
- **效果**：复杂推理任务正确率显著提升
- **应用**：数学题、逻辑推理、多步骤决策

### 3.4 结构化输出
- **目标**：让模型输出 JSON、表格等结构化格式
- **方法**：
  - 在 prompt 里明确要求格式
  - 用 Few-shot 给例子
  - 用 OpenAI 的 function calling
- **重要性**：后端能直接解析，不需要正则提取

### 3.5 Prompt 模板与变量
- **原理**：把可变部分（用户问题、检索文档）插进固定模板
- **工程化基础**：代码维护、版本管理、A/B 测试
- **示例**：
  ```python
  template = """
  参考文档：{documents}
  
  用户问题：{question}
  
  请根据参考文档回答。
  """
  ```

### 3.6 Prompt 版本管理与 A/B 测试
- **为什么需要**：Prompt 改一个字，效果可能天差地别
- **版本管理**：
  - 给每个版本编号（v1, v2, v3）
  - 记录修改内容和理由
  - 记录评测结果
- **A/B 测试**：
  - 同一批测试题，分别用 v1 和 v2 跑
  - 对比准确率、用户满意度
- **本项目实现**：Prompt 版本管理页面

## 第 4 层：RAG（检索增强生成）

### 4.1 为什么需要 RAG
- **问题 1**：模型不知道你公司的私有知识
- **问题 2**：模型不知道最新信息（训练数据有截止日期）
- **问题 3**：模型爱编（幻觉）
- **解决方案**：RAG = 先去知识库**查**相关资料 → 把资料**塞给**模型 → 让它**照着答**

### 4.2 RAG 的工作流程
1. **离线阶段**：
   - 文档切分（Chunking）
   - 向量化（Embedding）
   - 存入向量数据库
2. **在线阶段**：
   - 用户提问
   - 查询向量化
   - 检索相似文档（Retrieval）
   - 把文档和问题一起给 LLM
   - LLM 生成答案（Generation）

### 4.3 Chunking（文档切分）
- **为什么要切**：长文档塞不进上下文、检索精度低
- **切分策略**：
  - 按固定长度（如 500 字）
  - 按段落
  - 按语义（更智能，但更复杂）
- **权衡**：
  - 切太大：塞不下、噪声多
  - 切太小：丢失上下文、语义不完整
- **本项目第一版**：按行切（>10 字）
- **本项目第二版（新项目）**：按段落 + 重叠切分

### 4.4 Embedding（向量化）⭐⭐⭐
- **核心思想**：把文字变成一串数字（向量），语义相近的文字向量也相近
- **例子**：
  - "手机配对" → [0.2, 0.8, 0.1, ...]
  - "蓝牙连接" → [0.3, 0.7, 0.15, ...] （向量相近）
  - "今天天气" → [0.9, 0.1, 0.05, ...] （向量很远）
- **语义检索的基础**：关键词匹配做不到的"理解同义词"
- **第一个项目的问题**：用 bigram 关键词匹配，"手机配对"和"蓝牙连接"匹配不上
- **本项目升级**：用 Embedding 实现真正的语义检索

### 4.5 向量数据库
- **作用**：专门存向量、做相似度搜索
- **常见选择**：
  - FAISS（Facebook，轻量级，适合本地）
  - Chroma（简单易用）
  - Milvus（生产级，支持分布式）
  - Pinecone（云服务，不需要自己部署）
- **本项目选择**：FAISS（免费、本地、够用）

### 4.6 相似度算法
- **余弦相似度**：向量夹角，最常用
- **欧氏距离**：向量距离
- **Jaccard 相似度**：集合交并比（你第一个项目用的）
- **点积**：向量点乘

### 4.7 召回（Recall）与重排（Rerank）
- **召回**：第一步，粗筛一批候选（如 top-20）
- **重排**：第二步，用更精的模型重新排序（如 top-3）
- **为什么分两步**：平衡效果和速度
- **本项目的 top_k**：就是"取前几条"

### 4.8 引用溯源（Citation）
- **作用**：把答案的依据指回原文
- **重要性**：让用户能核对，提升可信度
- **实现方式**：
  - 在答案后加"[1][2]"引用标记
  - 返回引用的原文段落
- **本项目实现**：展示引用文档列表

## 第 5 层：幻觉与可信度

### 5.1 什么是幻觉（Hallucination）
- **定义**：模型一本正经地胡编
- **原因**：模型是概率接话机器，不"知道"事实
- **危害**：AI 产品最大的风险点，尤其在医疗、法律、金融等领域

### 5.2 降低幻觉的手段
1. **RAG 给依据**：让模型"照着文档说"
2. **温度调低**：减少随机性
3. **System prompt 约束**：明确告诉它"不知道就说不知道"
4. **引用溯源**：逼着它给出依据
5. **人工审核**：高风险场景，人工确认后再输出

### 5.3 幻觉检测
- **方案 A（你第一个项目）**：
  - 提取 AI 回答的关键词
  - 检查这些关键词有多少能在引用文档中找到
  - 准确率 = 有支撑的关键词数 / 总关键词数
  - **亮点**：不需要标准答案，自动检测幻觉风险
- **方案 B（本项目补充）**：
  - LLM-as-judge：用另一个大模型给答案打分
  - 判断"答案是否与文档一致"
- **方案 C（生产环境）**：
  - 人工标注 + 模型辅助

### 5.4 可信度指标
- **引用率**：答案中有多少比例来自文档
- **拒答率**：遇到不知道的问题，AI 拒答的比例
- **采纳率**：用户认为答案有用的比例
- **人工接管率**：AI 答不上，转人工的比例

## 第 6 层：Agent 与工具调用

### 6.1 Function Calling（函数调用）
- **原理**：让模型在需要时"调用一个工具"
- **应用场景**：
  - 查天气：调用天气 API
  - 查数据库：生成 SQL 查询
  - 下单：调用订单系统接口
- **让 AI 从"会聊"变成"会干活"**

### 6.2 ReAct（推理+行动）
- **流程**：思考 → 行动 → 观察 → 思考 → ...
- **让模型自己决定**：下一步用什么工具、什么时候停
- **应用**：复杂任务的自动化

### 6.3 多轮对话与记忆
- **短期记忆**：把历史对话塞进上下文
- **长期记忆**：存数据库，需要时检索出来
- **挑战**：上下文窗口有限，怎么取舍历史信息

### 6.4 多 Agent 协作
- **思想**：多个 AI 角色分工
- **示例**：
  - 规划者：拆解任务
  - 执行者：完成子任务
  - 审查者：检查结果
- **应用**：复杂系统、代码生成、自动化测试

## 第 7 层：成本、性能与稳定性

### 7.1 Token 计费
- **计费模式**：
  - 输入 token（prompt）：便宜
  - 输出 token（completion）：贵（通常是输入的 2～5 倍）
- **成本估算公式**：
  ```
  单次成本 = (输入 tokens × 输入单价 + 输出 tokens × 输出单价) / 1000
  ```
- **产品经理必备技能**：能算"这功能一个月烧多少钱"

### 7.2 延迟（Latency）
- **TTFT（Time To First Token）**：首字延迟，影响体验
- **总耗时**：生成全部答案的时间
- **优化手段**：
  - 流式输出（降低 TTFT）
  - 缓存常见问题
  - 用小模型（降低延迟）

### 7.3 限流（Rate Limit）
- **RPM（Requests Per Minute）**：每分钟请求数上限
- **TPM（Tokens Per Minute）**：每分钟 token 数上限
- **触发后的处理**：
  - 429 错误
  - 指数退避重试（你第一个项目实现了）
  - 队列缓冲

### 7.4 缓存
- **策略**：相同问题缓存答案
- **收益**：省钱、省时间
- **挑战**：问题的"相同"怎么判断（完全匹配 vs 语义相似）

## 第 8 层：产品经理核心能力

### 8.1 需求与 PRD
- **用户研究**：用户是谁、痛点是什么
- **为什么用 AI 解**：AI 的优势在哪、边界在哪
- **PRD 结构**：
  - 产品背景
  - 用户画像
  - 核心功能
  - 技术方案
  - 指标体系
  - 里程碑

### 8.2 指标体系 ⭐⭐⭐
- **北极星指标**：最核心的一个指标
- **转化漏斗**：用户从进入到完成目标的各环节转化率
- **AI 特有指标**：
  - **准确率**：答对 / 总问题
  - **召回率**：能答 / 该答
  - **拒答率**：主动拒答 / 总问题
  - **采纳率**：用户点赞 / 总回答
  - **人工接管率**：转人工 / 总问题
  - **平均解决时长**：从提问到问题解决的时间
  - **成本**：单次交互成本

### 8.3 数据埋点
- **埋什么**：
  - 用户提问内容
  - AI 回答内容
  - 用户是否点"有用/没用"
  - 是否追问
  - 是否放弃
  - 检索到的文档
  - 参数设置（温度、模型等）
- **为什么重要**：没有埋点就没有迭代依据
- **本项目实现**：交互日志 + 运营看板

### 8.4 评测体系
- **离线评测**：
  - 准备标准测试集
  - 跑自动化评测
  - 对比不同版本
- **在线评测**：
  - 真实流量 A/B 测试
  - 用户反馈收集
- **LLM-as-judge**：
  - 用大模型给大模型的答案打分
  - 快速、便宜、可规模化
- **人工标注**：
  - 金标准
  - 慢、贵，但最准

### 8.5 竞品分析与选型决策
- **什么场景用 Prompt**：
  - 零样本任务
  - 快速验证
  - 成本最低
- **什么时候上 RAG**：
  - 需要私有知识
  - 需要最新信息
  - 幻觉风险高
- **什么时候才微调**：
  - Prompt + RAG 都解决不了
  - 有大量标注数据
  - 有预算和时间

### 8.6 合规与安全
- **数据隐私**：用户数据是否会被模型提供商看到
- **内容安全**：如何防止 AI 输出违规内容
- **Prompt 注入攻击**：恶意用户通过 prompt 绕过规则
- **风险缓解**：
  - 输入过滤
  - 输出审核
  - 权限控制
  - 日志审计

## 附录：常见面试问题

### 技术类
1. **RAG 和微调的区别是什么？**
   - RAG：外挂知识库，灵活、便宜
   - 微调：改模型内部，性能好、贵

2. **怎么降低幻觉？**
   - RAG、低温度、prompt 约束、引用溯源

3. **Embedding 是什么？**
   - 把文字变成向量，语义相近的向量也相近

4. **Temperature 设多少合适？**
   - 看场景，客服 0.1～0.3，创意 0.7～0.9

### 产品类
1. **AI 产品的核心指标是什么？**
   - 准确率、采纳率、拒答率、成本、延迟

2. **怎么评估一个 AI 产品的效果？**
   - 离线：测试集评测
   - 在线：用户反馈 + A/B 测试

3. **什么时候用 RAG，什么时候微调？**
   - RAG：需要私有知识、需要更新、成本敏感
   - 微调：需要改变模型行为、有大量数据

4. **怎么控制 AI 产品的成本？**
   - 缓存、用小模型、控制输出长度、限流
