# AI 产品经理基础知识

## 1. 大语言模型（LLM）基础

### 1.1 什么是 LLM
大语言模型（Large Language Model，LLM）是一种基于深度学习的自然语言处理模型。它通过学习海量文本数据，掌握了语言的统计规律和语义关系。

LLM 的核心能力包括：
- 文本生成：根据输入生成连贯的文本
- 问答：回答用户的问题
- 摘要：提取文本的核心内容
- 翻译：在不同语言之间转换
- 代码生成：根据描述生成代码

### 1.2 Token 是什么
Token 是 LLM 处理文本的最小单位。在英文中，一个单词通常是 1 个 token；在中文中，1 个汉字通常对应 1-2 个 token。

Token 很重要，因为：
- 模型的计费按 token 计算
- 上下文窗口的限制也是以 token 为单位
- 不同模型对 token 的定义可能略有不同

例如："你好世界" 大约是 6-8 个 tokens。

### 1.3 上下文窗口
上下文窗口（Context Window）是指模型一次能"看到"的 token 数量上限。

常见模型的窗口大小：
- GPT-3.5: 4K 或 16K tokens
- GPT-4: 8K、32K 或 128K tokens
- Claude: 100K 或 200K tokens

上下文窗口的限制意味着：
- 不能一次性输入超长的文档
- 多轮对话的历史记录会占用窗口
- 这是 RAG 技术存在的原因之一

## 2. API 调用参数

### 2.1 Temperature（温度）
Temperature 控制模型输出的随机性和创造性。

取值范围：0 到 2（常用 0 到 1）

效果：
- Temperature = 0：输出几乎确定，适合需要稳定答案的场景（如客服、信息抽取）
- Temperature = 0.3：略有变化，但保持保守，适合智能客服
- Temperature = 0.7-1.0：输出更有创意，适合文案创作、头脑风暴
- Temperature > 1.0：非常随机，可能产生不连贯的内容

应用建议：
- 客服机器人：使用 0.1-0.3
- 创意写作：使用 0.7-0.9
- 代码生成：使用 0.2

### 2.2 Top-p（核采样）
Top-p 是另一种控制随机性的方式，也叫核采样（Nucleus Sampling）。

工作原理：模型只在"累计概率达到 p"的候选词中选择。

例如，top_p = 0.9 表示只考虑累计概率达到 90% 的词。

与 Temperature 的关系：
- 两者作用类似，一般只调一个
- 不建议同时大幅调整两个参数

推荐设置：
- 确定性任务：top_p = 0.9，temperature = 0.1
- 创造性任务：top_p = 0.95，temperature = 0.8

### 2.3 Max Tokens
Max Tokens 限制模型输出的最大长度（以 token 为单位）。

影响：
- 设置太小：回答可能被截断
- 设置太大：浪费成本和时间

使用建议：
- 根据实际需求设置合理上限
- 简单问答：200-500 tokens
- 详细解释：500-1000 tokens
- 长文生成：1000-2000 tokens

## 3. RAG（检索增强生成）

### 3.1 为什么需要 RAG
LLM 虽然强大，但存在三个问题：
1. 不知道私有知识：企业内部文档、产品手册等
2. 不知道最新信息：模型的训练数据有截止日期
3. 容易产生幻觉：编造不存在的信息

RAG 通过"先检索、再生成"的方式解决这些问题。

### 3.2 RAG 的工作流程
离线阶段：
1. 将知识库文档切分成小段落
2. 用 Embedding 模型把每段转换为向量
3. 将向量存入向量数据库

在线阶段：
1. 用户提问
2. 将问题转换为向量
3. 在向量数据库中检索最相关的文档段落
4. 将文档和问题一起发给 LLM
5. LLM 根据文档生成答案

### 3.3 Embedding（向量化）
Embedding 是将文本转换为数字向量的技术。

核心思想：语义相近的文本，其向量也相近。

例如：
- "手机配对" 和 "蓝牙连接" 的向量会很接近
- "手机配对" 和 "今天天气" 的向量会很远

这使得系统能够理解同义词和相关概念，而不仅仅是关键词匹配。

### 3.4 向量数据库
向量数据库专门用于存储和检索向量。

常见的向量数据库：
- FAISS：Facebook 开源，轻量级，适合本地使用
- Chroma：简单易用，适合快速原型
- Milvus：生产级，支持分布式部署
- Pinecone：云服务，不需要自己维护

选择建议：
- 学习项目：使用 FAISS
- 中小规模应用：使用 Chroma
- 大规模生产环境：使用 Milvus 或 Pinecone

## 4. 幻觉与可信度

### 4.1 什么是幻觉
幻觉（Hallucination）是指 LLM 一本正经地编造不存在的信息。

常见的幻觉类型：
- 编造事实：虚构的统计数据、日期、人名
- 编造细节：文档中没有的具体操作步骤
- 自相矛盾：前后说法不一致

### 4.2 如何降低幻觉
主要方法：
1. 使用 RAG：让 AI 根据文档回答，而不是凭空生成
2. 降低 Temperature：减少随机性
3. 优化 Prompt：明确告诉 AI "不知道就说不知道"
4. 引用溯源：要求 AI 给出信息来源
5. 人工审核：高风险场景必须人工确认

### 4.3 幻觉检测方法
方案 A（关键词匹配法）：
- 提取 AI 回答中的关键词
- 检查这些关键词有多少能在原文档中找到
- 计算支撑率：有支撑的关键词数 / 总关键词数
- 支撑率越高，幻觉风险越低

方案 B（LLM-as-judge）：
- 用另一个 LLM 评判答案是否与文档一致
- 快速、可规模化
- 但评判者本身也可能出错

方案 C（人工标注）：
- 最准确
- 但成本高、速度慢

## 5. Prompt 工程

### 5.1 什么是 Prompt
Prompt 是给 LLM 的指令和上下文。好的 Prompt 能显著提升输出质量。

Prompt 的组成：
- System Prompt：定义 AI 的身份和规则
- User Prompt：用户的具体问题或任务
- Few-shot Examples：给几个示例帮助 AI 理解

### 5.2 Prompt 优化技巧
1. 明确角色：告诉 AI 它是什么（如"你是一个专业的客服助手"）
2. 清晰指令：用祈使句，如"请根据文档回答"
3. 设定边界：告诉 AI 什么不能做
4. 结构化输出：要求特定格式（如 JSON、列表）
5. 提供示例：Few-shot learning 效果显著

### 5.3 Prompt 版本管理
为什么要版本管理：
- Prompt 改一个字，效果可能天差地别
- 需要记录每次改动和原因
- 通过 A/B 测试验证改进效果

最佳实践：
- 每个版本有清晰的命名和描述
- 记录改动原因和预期效果
- 跑标准测试集对比效果
- 线上灰度验证再全量上线

## 6. 评测与指标

### 6.1 AI 产品的核心指标
北极星指标：
- 最重要的一个指标
- 通常是"问题解决率"或"用户采纳率"

一级指标：
- 交互量：使用深度
- 采纳率：产品质量
- 响应时间：性能体验
- 成本：商业可行性

二级指标（AI 特有）：
- 准确率：答对的比例
- 召回率：能答的比例
- 拒答率：主动拒答的比例
- 幻觉率：编造内容的比例
- 人工接管率：转人工的比例

### 6.2 离线评测 vs 在线评测
离线评测：
- 在开发环境用标准测试集评测
- 快速、可控、可复现
- 但测试集可能不够全面

在线评测：
- 在真实流量上 A/B 测试
- 真实、全面
- 但慢、影响用户

最佳实践：先离线验证方向，再小流量灰度，最后全量上线。

### 6.3 LLM-as-judge
用一个 LLM 评判另一个 LLM 的输出质量。

优点：
- 自动化、可规模化
- 不需要人工标注
- 可以评估主观指标（如"回答是否清晰"）

缺点：
- 评判者本身也可能出错
- 增加成本
- 评分标准可能不稳定

适用场景：
- 快速迭代阶段
- 辅助人工评测
- 大规模评测

## 7. 成本与优化

### 7.1 成本构成
LLM API 的成本主要来自 Token 消耗：
- 输入 Token（Prompt）：相对便宜
- 输出 Token（Completion）：通常是输入的 2-5 倍价格

成本计算公式：
成本 = (输入 tokens × 输入单价 + 输出 tokens × 输出单价) / 1000

### 7.2 成本优化方法
1. 缓存：相同问题缓存答案
2. 限制输出长度：合理设置 max_tokens
3. 选择合适的模型：不是所有任务都需要最强的模型
4. 批量处理：减少 API 调用次数
5. 压缩 Prompt：去掉不必要的内容

### 7.3 性能优化
提升响应速度的方法：
1. 流式输出：降低首字延迟
2. 优化检索：使用更快的向量索引
3. 并行处理：同时处理多个请求
4. 选择更快的模型：小模型通常更快
5. 缓存：减少重复计算
