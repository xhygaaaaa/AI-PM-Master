## 一、产品背景

### 1.1 市场机会
- **企业知识管理痛点**：企业内部文档散落各处（Wiki、共享盘、聊天记录），员工找不到、找不全、找不快
- **客服成本高**：人工客服重复回答相同问题，成本高、效率低、夜间无人值守
- **AI 技术成熟**：大语言模型（LLM）+ RAG 技术已经能够实现高质量的知识问答

### 1.2 产品定位
一个**面向产品经理学习的 AI 问答系统**，通过完整的技术实现 + 产品运营能力展示，帮助 AI 产品经理掌握从技术到业务的全链路能力。

### 1.3 目标用户
- **主要用户**：准备投 AI 产品经理岗位的求职者
- **次要用户**：想要理解 AI 产品全流程的产品经理、开发工程师

## 四、技术方案

### 4.1 技术架构

```
┌─────────────────────────────────────────────────────┐
│                   前端展示层                           │
│              Streamlit Multi-Page App                 │
├─────────────────────────────────────────────────────┤
│                   业务逻辑层                           │
│  ├─ vector_rag.py         语义检索引擎                │
│  ├─ llm_client.py         LLM 调用封装                │
│  ├─ prompt_manager.py     Prompt 版本管理             │
│  ├─ evaluator.py          评测逻辑                    │
│  ├─ hallucination_detector.py  幻觉检测              │
│  └─ analytics.py          数据统计                    │
├─────────────────────────────────────────────────────┤
│                  微服务层（可选）                       │
│         FastAPI Evaluation Service                    │
│              LLM-as-judge 评测接口                     │
├─────────────────────────────────────────────────────┤
│                   数据存储层                           │
│  ├─ FAISS 向量索引       （内存/磁盘）                 │
│  ├─ JSONL 交互日志       （data/logs/）               │
│  ├─ JSON Prompt 版本     （data/prompts/）            │
│  └─ CSV 评测结果         （data/test_sets/）          │
├─────────────────────────────────────────────────────┤
│                   外部依赖                             │
│  ├─ OpenAI API / 通义 / DeepSeek  (LLM 服务)         │
│  └─ sentence-transformers       (本地 Embedding)      │
└─────────────────────────────────────────────────────┘
```

### 4.2 技术选型

| 技术选项 | 选型 | 理由 |
|---------|------|------|
| 前端框架 | Streamlit | 快速开发、适合展示、无需前端技能 |
| 后端框架 | FastAPI | 轻量、异步、自动生成文档 |
| LLM 调用 | OpenAI SDK | 兼容多家大模型（OpenAI / 通义 / DeepSeek / MiniMax） |
| Embedding 模型 | sentence-transformers | 开源、本地运行、中文支持好 |
| 向量数据库 | FAISS | 免费、轻量、适合中小规模 |
| 数据存储 | JSONL / CSV / JSON | 简单、可读、易调试 |
| 部署方式 | Docker Compose | 一键启动、环境隔离 |

### 4.3 数据流

```
用户提问
   ↓
[前端] Streamlit 接收输入
   ↓
[业务层] 问题向量化 (sentence-transformers)
   ↓
[存储层] FAISS 检索 top-k 相似文档
   ↓
[业务层] 构建 Prompt (模板 + 检索文档 + 用户问题)
   ↓
[外部] 调用 LLM API (带参数: temperature, max_tokens 等)
   ↓
[业务层] 解析 LLM 回答
   ↓
[业务层] 幻觉检测 (关键词匹配原文档)
   ↓
[存储层] 记录交互日志 (JSONL)
   ↓
[前端] 展示答案 + 引用 + 幻觉风险提示
```

## 五、指标体系

### 5.1 北极星指标
**用户学习完成率** = 完成 6 个核心模块操作的用户数 / 总访问用户数

### 5.2 一级指标

| 指标类别 | 指标名称 | 定义 | 目标值 |
|---------|---------|------|--------|
| 使用深度 | 平均使用模块数 | 用户平均使用的功能模块数量 | ≥ 3 个 |
| 学习效果 | 知识点覆盖率 | 用户实际操作覆盖的知识点 / 总知识点 | ≥ 60% |
| 产品质量 | AI 回答采纳率 | 用户点"有用"的次数 / 总回答次数 | ≥ 70% |
| 技术性能 | 平均响应时间 | 从提问到返回答案的时间 | ≤ 5 秒 |

### 5.3 二级指标（AI 特有）

| 指标名称 | 定义 | 计算公式 | 说明 |
|---------|------|---------|------|
| 准确率 | 答对的问题 / 总问题 | Correct / Total | 需要标准测试集 |
| 召回率 | 能答的问题 / 该答的问题 | Answered / Should_Answer | 衡量"不拒答该答的" |
| 拒答率 | 主动拒答 / 总问题 | Refused / Total | 合理的拒答是好事 |
| 幻觉率 | 检测到幻觉 / 总回答 | Hallucination / Total | 用方案 A 算法检测 |
| 平均 Token 消耗 | 单次交互的 Token 数 | Avg(input_tokens + output_tokens) | 成本控制 |
| 单次成本 | 单次交互的费用 | Token 数 × 单价 | 财务指标 |

### 5.4 转化漏斗

```
100 人访问首页
  ↓ (80%)
80 人完成第一次提问
  ↓ (60%)
48 人尝试调节参数
  ↓ (50%)
24 人使用 Prompt 版本管理
  ↓ (40%)
10 人运行评测中心
  ↓ (80%)
8 人查看运营看板

核心转化率 = 8 / 100 = 8%
目标：提升到 15%
```

## 六、用户旅程

### 场景 1：快速体验（5 分钟）
1. 用户打开应用
2. 在首页提问："什么是 Temperature 参数？"

1. 用户进入"参数调优实验室"
2. 调节 temperature 从 0 → 1，观察回答变化
3. 理解：温度越高越随机
4. 进入"Prompt 版本管理"
5. 保存两个版本的 Prompt，运行 A/B 对比
6. 理解：Prompt 版本管理的重要性
7. 进入"评测中心"
8. 上传测试集，运行批量评测
9. 查看准确率报告
10. 理解：离线评测的流程

### 场景 3：项目总结（1 小时）
1. 用户阅读 PRD 文档
2. 查看运营看板，理解数据指标
3. 阅读技术文档，理解代码架构

## 七、风险与挑战

### 7.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| API 限流 | 评测失败 | 高 | 指数退避重试 + 控制并发 |
| Embedding 模型下载慢 | 首次启动慢 | 中 | 提供国内镜像 + 预下载脚本 |
| 向量索引占用内存大 | 低配机器跑不动 | 中 | 使用 FAISS 的磁盘索引模式 |
| LLM 幻觉 | 答案错误 | 高 | 幻觉检测 + 引用溯源 + 用户反馈 |

### 7.2 产品风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 用户学习曲线陡 | 放弃使用 | 中 | 详细的新手引导 + 代码注释 |
| 知识点太多记不住 | 学习效果差 | 高 | 知识点文档 + 实操演示 + 总结清单 |
| 没有真实业务场景 | 缺乏代入感 | 低 | PRD 设计真实用户画像和场景 |

### 7.3 成本风险

| 成本项 | 估算（月） | 说明 |
|--------|-----------|------|
| LLM API 调用 | ¥50～200 | 取决于使用量，可用免费额度 |
| Embedding 模型 | ¥0 | 本地运行，免费 |
| 服务器 / 云部署 | ¥0～50 | Streamlit Community Cloud 免费 |
| 总计 | ¥50～250 | 个人学习项目成本可控 |

## 十一、FAQ

### Q1：为什么要做 FastAPI 微服务？
A：展示"前后端协作"能力，体现后端简历。实际上 LLM-as-judge 放在 Streamlit 里也行，但拆出来更有架构感。

### Q2：为什么选 FAISS 而不是 Chroma？
A：FAISS 更轻量、性能更好、部署更简单。Chroma 功能更全但对新手来说过于复杂。

### Q3：为什么不做微调？
A：微调需要大量数据 + GPU + 长时间，超出个人学习项目的范围。RAG 已经能解决 90% 的问题。

### Q4：为什么不做 Agent / Function Calling？
A：核心模块已经覆盖 AI PM 的必备能力，Agent 是加分项。Phase 3 可选加入。

### Q5：项目能写进简历吗？
A：完全可以。建议写法：
- "独立设计并开发 AI 智能问答系统，涵盖 RAG、Prompt 工程、评测体系"
- "实现语义检索（Embedding + FAISS）、参数调优、Prompt 版本管理等功能"
- "建立完整的 AI 产品指标体系（准确率、采纳率、成本等）"
- "使用 LLM-as-judge 实现自动化评测，提升评测效率 80%"
