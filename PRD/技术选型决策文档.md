# 技术选型决策文档：Prompt vs RAG vs 微调

> **本文档目标**：帮 AI 产品经理建立"什么时候用什么技术"的决策框架。
> 这是 AI PM 最核心的战略能力之一，面试高频考点。

---

## 一、三种技术方案速览

做 AI 产品，让模型"懂业务"主要有三条路，成本和复杂度从低到高：

| 方案 | 一句话解释 | 成本 | 上线速度 | 适合阶段 |
|------|-----------|------|---------|---------|
| **Prompt 工程** | 用提示词引导模型 | 最低 | 最快（几小时） | 快速验证 |
| **RAG** | 外挂知识库 | 中 | 较快（几天） | 大多数知识类产品 |
| **微调** | 改模型本身 | 最高 | 最慢（几周） | 特殊行为定制 |

**黄金法则**：从便宜的开始试，不够用再往上加。**90% 的需求 Prompt + RAG 就能搞定，不要一上来就想微调。**

---

## 二、方案详解

### 2.1 Prompt 工程

**做什么**：通过精心设计的提示词（System Prompt + Few-shot 示例），引导模型按你想要的方式回答。

**优点**：
- 零成本启动，改个字符串就行
- 迭代极快，改完立刻见效
- 不需要数据、不需要训练、不需要 GPU

**缺点**：
- 受限于模型本身的知识（不知道你的私有数据）
- Prompt 太长会增加每次调用的成本
- 对模型能力依赖大，弱模型怎么调都不行

**适用场景**：
- 文本分类、摘要、改写、翻译
- 格式转换（自然语言转 JSON）
- 快速验证一个 AI 想法能不能行

**本项目的体现**：[Prompt 版本管理](../pages/2_Prompt版本管理.py) 模块，3 个版本（基础/严格/友好）就是纯 Prompt 工程。

---

### 2.2 RAG（检索增强生成）

**做什么**：把私有知识存进向量库，回答前先检索相关内容，塞进 Prompt 让模型"照着答"。

**优点**：
- 让模型掌握私有知识（公司文档、产品手册）
- 知识可实时更新（改文档即可，不用重训）
- 大幅降低幻觉（有据可依）
- 可溯源（能指出答案来自哪份文档）

**缺点**：
- 比纯 Prompt 复杂（要搭向量库、做切分、调检索）
- 检索质量直接决定回答质量（检索错了，回答必错）
- 每次调用的 Prompt 更长 = 成本更高

**适用场景**：
- 企业知识库问答、智能客服
- 文档助手、法律/医疗知识查询
- 任何"需要基于特定资料回答"的场景

**本项目的体现**：[语义检索引擎](../src/vector_rag.py) + [智能问答主页](../main.py)，这是整个项目的核心。

---

### 2.3 微调（Fine-tuning）

**做什么**：用你的标注数据，对模型再训练一遍，改变模型的内在行为。

**优点**：
- 能改变模型的"性格"和输出风格（如固定的格式、语气）
- 能学会 Prompt 教不会的复杂模式
- 推理时 Prompt 可以更短（行为已内化）

**缺点**：
- 贵：需要 GPU、需要大量标注数据（通常上千条起）
- 慢：数据准备 + 训练 + 评估，以周计
- 死板：知识更新要重新训练
- 容易"学坏"：数据质量差会让模型变笨

**适用场景**：
- 需要固定的、特殊的输出风格（如特定的医疗报告格式）
- 有大量高质量标注数据
- Prompt + RAG 都试过了，效果还是不够
- 对推理延迟/成本极度敏感（用小模型微调替代大模型）

**本项目**：未实现微调（超出个人学习项目范围）。但你要能讲清楚"什么时候才需要它"。

## 三、决策流程图

```
需求来了
   ↓
问题1：模型本身的通用知识够不够回答？
   ├─ 够 → 用 Prompt 工程（最简单）
   └─ 不够，需要私有/最新知识 ↓
   
问题2：私有知识能整理成文档吗？
   ├─ 能 → 用 RAG（外挂知识库）
   └─ 不能，是行为/风格问题 ↓
   
问题3：Prompt 反复调试还是达不到要求？
   ├─ 是，且有大量标注数据 → 考虑微调
   └─ 否 → 回到 Prompt + RAG 继续优化
```

**关键提醒**：微调是最后选项，不是首选。很多人误以为"AI 不准就要微调"，其实 90% 的情况是 Prompt 没写好或 RAG 没搭好。

## 四、组合拳：真实产品往往是叠加的

真实产品很少只用一种技术，通常是组合：

| 组合 | 典型产品 | 说明 |
|------|---------|------|
| Prompt + RAG | 企业客服、文档助手 | 最常见，本项目就是这个 |
| Prompt + Function Calling | 智能助手、Copilot | 能查实时数据、执行操作 |
| RAG + Function Calling | 高级 Agent | 既懂知识又能干活（本项目演示页有雏形）|
| RAG + 微调 | 垂直领域专家系统 | 微调定制风格 + RAG 提供知识 |

**本项目的组合**：Prompt 工程（版本管理）+ RAG（语义检索）+ Function Calling（工具调用演示），覆盖了主流的前三种。

## 五、竞品/工具选型对比

### 5.1 开发框架选型

| 工具 | 定位 | 优点 | 缺点 | 适合谁 |
|------|------|------|------|--------|
| **LangChain** | 全能框架 | 功能全、生态大 | 重、抽象层多、学习曲线陡 | 复杂应用 |
| **LlamaIndex** | RAG 专精 | RAG 做得深 | 偏 RAG，其他弱 | RAG 重度 |
| **Dify / Coze** | 低代码平台 | 拖拽即用、快 | 灵活性差、难定制 | 快速验证、非技术 |
| **自己写**（本项目）| 手搓 | 完全可控、理解透彻 | 要自己造轮子 | 学习、轻量项目 |

**本项目为什么手搓**：学习目的。手写一遍 Embedding、向量检索、Function Calling，比用框架黑盒调用理解深得多。面试时能讲清楚底层，这是用 Dify 点点点学不到的。

### 5.2 向量数据库选型

| 数据库 | 优点 | 缺点 | 适合规模 |
|--------|------|------|---------|
| **FAISS**（本项目）| 轻量、快、免费、本地 | 无持久化服务、功能基础 | 中小（百万级以下）|
| **Chroma** | 简单易用、自带持久化 | 性能一般 | 中小 |
| **Milvus** | 生产级、分布式 | 重、运维复杂 | 大规模 |
| **Pinecone** | 云托管、省心 | 收费、数据在别人服务器 | 不想自己运维 |

**本项目为什么用 FAISS**：学习项目数据量小，FAISS 轻量够用，不引入额外的服务依赖，部署简单。

### 5.3 大模型选型

| 模型 | 优点 | 缺点 | 适合场景 |
|------|------|------|---------|
| **GPT-4** | 最强 | 最贵、需翻墙 | 高难度任务 |
| **通义千问** | 中文好、阿里生态 | 略贵 | 中文企业应用 |
| **DeepSeek** | 极便宜、效果好 | 品牌新 | 成本敏感场景 |
| **MiniMax** | 有免费额度 | 能力中等 | 学习、原型 |
| **本地 Ollama** | 免费、数据不出门 | 要好硬件、效果弱 | 隐私敏感、离线 |

**选型方法论**：用本项目的 [成本与延迟仪表盘](../pages/5_成本与延迟仪表盘.py) 算成本 + [评测中心](../pages/3_评测中心.py) 测效果，两者结合做决策。不要凭感觉选模型。

## 六、面试话术模板

**Q：什么时候用 RAG，什么时候用微调？**

A：我的判断逻辑是这样的——首先看是知识问题还是行为问题。如果是"模型不知道某些信息"，比如公司的产品文档，那用 RAG，把文档存进向量库，回答前检索出来。RAG 的好处是知识可以随时更新，改文档就行，不用重训。

只有当 Prompt 和 RAG 都解决不了，比如需要模型有特定的输出风格、或者需要它学会一种复杂的模式，而且我手里有足够的标注数据，我才会考虑微调。因为微调成本高、周期长、知识更新还得重训。

实际上我做的项目里，90% 的需求 Prompt + RAG 就够了。我自己实现过一个完整的 RAG 系统，用 Embedding + FAISS 做语义检索，这个过程让我对什么时候该上什么技术有了很具体的判断。

**Q：你怎么选大模型？**

A：我不会凭感觉选，而是用数据。我会准备一个标准测试集，把候选模型都跑一遍，看准确率；同时算清楚每个模型的单次成本和月度成本。然后结合效果和成本做决策——如果便宜的模型准确率够用，就没必要为贵模型多付好几倍的钱。我项目里专门做了成本仪表盘和评测中心就是干这个的。

## 七、本项目技术选型总结

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 开发方式 | 手搓（不用 LangChain）| 学习目的，理解底层 |
| 知识方案 | RAG | 需要私有知识、可更新、降幻觉 |
| 向量库 | FAISS | 轻量、免费、够用 |
| Embedding | sentence-transformers | 本地、免费、中文支持好 |
| 前端 | Streamlit | 快速开发、适合演示 |
| 后端 | FastAPI 微服务 | 体现前后端分离架构 |
| 部署 | Docker Compose | 一键启动、环境隔离 |

每一个选择背后都有"为什么"，这才是 PM 的价值——不是会用工具，而是知道为什么用这个工具。
